# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Hermes Agent is a self-improving AI agent built by Nous Research. It wraps any LLM provider (OpenAI, Anthropic, OpenRouter, etc.) with a learning loop: persistent memory, auto-generated skills, multi-platform messaging gateway (Telegram, Discord, Slack, WhatsApp, Signal), and 40+ built-in tools.

## Development Setup

```bash
git clone --recurse-submodules https://github.com/NousResearch/hermes-agent.git
cd hermes-agent
uv venv venv --python 3.11
source venv/bin/activate
uv pip install -e ".[all,dev]"
mkdir -p ~/.hermes/{cron,sessions,logs,memories,skills}
```

For RL training integration (optional):
```bash
git submodule update --init tinker-atropos
uv pip install -e "./tinker-atropos"
```

## Commands

```bash
# Run the CLI
hermes

# Run all tests (parallel, skips integration by default)
python -m pytest tests/ -q

# Run a single test file
python -m pytest tests/agent/test_prompt_builder.py -q

# Run a single test
python -m pytest tests/agent/test_prompt_builder.py::TestClass::test_name -q

# Include integration tests (requires external services/API keys)
python -m pytest tests/ -m integration -q

# Direct agent runner (bypasses CLI)
python run_agent.py --help
```

## Architecture

### Entry Points

| Script | Purpose |
|--------|---------|
| `hermes_cli/main.py` | Primary `hermes` CLI — dispatches all subcommands |
| `run_agent.py` | `AIAgent` class (9,900 lines) — the core agent loop |
| `gateway/run.py` | `GatewayRunner` — multi-platform messaging event loop |
| `acp_adapter/entry.py` | Agent Client Protocol adapter |

### Agent Loop (`run_agent.py`)

`AIAgent` is the orchestrator: it receives a user message, sends it (with system prompt) to the LLM, receives tool calls, executes tools (parallelizing where paths don't conflict), streams results back, and iterates until the model signals completion or `max_iterations` (default 90) is reached. Context compression triggers automatically near token limits.

### Tools & Toolsets (`tools/`, `toolsets.py`)

- `tools/registry.py` — central `TOOL_REGISTRY` mapping names to tool instances
- Each tool returns a `ToolResult(content, error, is_error)` struct
- `toolsets.py` defines composable named groups (`"web"`, `"code"`, `"research"`, etc.) used to restrict tool availability per platform or session
- **Skill vs Tool rule**: use a Skill (YAML) to wrap an existing CLI/API; only build a Tool when you need native integration (streaming, binary data, browser sessions, API key management)

### Gateway & Platform Adapters (`gateway/`)

```
GatewayRunner (gateway/run.py)
├── BasePlatformAdapter  ← abstract base in gateway/platforms/
│   ├── TelegramAdapter, DiscordAdapter, SlackAdapter
│   ├── WhatsAppAdapter, SignalAdapter, MatrixAdapter
│   └── [15 more adapters]
├── SessionStore         ← persistent per-user conversations
├── DeliveryRouter       ← routes cron/scheduled messages
└── PairingStore         ← DM auth for messaging platforms
```

Each adapter handles auth, polling/webhooks, media caching, and command parsing. Platform adapters are gated on config availability (e.g., `TELEGRAM_TOKEN` must be set).

### Agent Sub-Packages (`agent/`)

Key modules used by `AIAgent`:

| Module | Role |
|--------|------|
| `anthropic_adapter.py` | Claude API integration |
| `prompt_builder.py` | System prompt construction |
| `memory_manager.py` | Injects user profile + session search into prompt |
| `context_compressor.py` | Token budget management |
| `error_classifier.py` | Error classification & provider failover |
| `credential_pool.py` | Multi-provider API key rotation |
| `model_metadata.py` | Model capability registry |

### Skills System (`skills/`, `optional-skills/`)

Skills are YAML files with structured instructions. They live in:
- `skills/` — 26+ bundled categories (github, research, productivity, …)
- `optional-skills/` — off-by-default, heavier dependencies
- `~/.hermes/skills/` — user-created skills
- Community skills at [agentskills.io](https://agentskills.io)

Skills are auto-generated by the agent after complex tasks and self-improve during use. `tools/skills_tool.py` handles the skill lifecycle; `tools/skill_manager_tool.py` handles create/update/delete.

### Memory (`agent/memory_manager.py`)

- FTS5-backed session search for cross-session recall
- `build_memory_context_block()` injects user profile + procedural memory into the system prompt
- Periodic nudges drive user modeling via Honcho dialectic refinement
- Storage: `~/.hermes/memories/` (JSON), `~/.hermes/sessions/` (snapshots)

### CLI (`hermes_cli/`)

41-module package. Key modules: `main.py` (dispatcher), `gateway.py` (gateway subcommand), `config.py`, `models.py`, `setup.py` (setup wizard), `skills_hub.py` (marketplace sync).

## Testing

- Tests live in `tests/` mirroring the source layout (`tests/agent/`, `tests/tools/`, `tests/gateway/`, `tests/cli/`)
- `tests/conftest.py` isolates `HERMES_HOME` to a temp directory so tests never touch `~/.hermes`
- Integration tests (require external APIs) are marked `@pytest.mark.integration` and skipped by default
- Parallel execution via `pytest-xdist` (`-n auto`) is the default

## Gotchas

- **Slash commands**: only `hermes_cli/commands.py` (`COMMAND_REGISTRY`) needs updating — CLI, gateway, Telegram, Slack, autocomplete, and help all derive from it automatically. No other file changes needed for aliases either.
- **Adding a tool** requires changes in 3 files: create `tools/your_tool.py`, register it in `toolsets.py`, and add it to `toolset_distributions.py`.
- **Skill slash commands** are injected as user messages (not system prompt) to preserve Anthropic prompt caching — see `agent/skill_commands.py`.
- **`[all]` extra excludes `[matrix]`** — `python-olm` (required by `matrix-nio[e2e]`) is broken on modern macOS (Clang 21+). Install manually: `pip install 'hermes-agent[matrix]'`.
